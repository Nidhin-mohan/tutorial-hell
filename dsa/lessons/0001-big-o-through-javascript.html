DSA for Interviews · Lesson 01

Big-O, taught through JavaScript you already write

The single most important DSA idea — and you can feel it in code you've used a hundred times.

⏱ ~12 min 🎯 The win: say a solution's Big-O out loud and mean it 🧩 Pattern seed: why we reach for Set & Map

Every coding interview, after you get a working answer, comes the question: "What's the time complexity?" Right now that question is scary. By the end of this lesson it won't be — because Big-O isn't academic. It's a label for something you already have intuition about.

Step 1 · The idea
Big-O = how the work grows as the input grows

Big-O does not measure milliseconds. It measures how the amount of work scales as your input n gets bigger. Two rules and you've got 90% of it:

  • Worst case. We describe the most work it could ever do.
  • Drop the noise. Constants and small terms don't matter at scale: O(2n + 5) is just O(n). We care about the shape of the growth, not the exact count.

The two shapes you'll meet constantly:

// O(1) — "constant": work does NOT grow with n.
// Grabbing one thing. Same effort for 10 items or 10 million.
const first = arr[0];

// O(n) — "linear": work grows in step with n.
// Touch every item once. 2x the items = 2x the work.
for (const x of arr) { total += x; }
Anchor it: you already know arr.map() visits every element. That's O(n). You already know arr[5] is instant. That's O(1). You've been doing Big-O intuitively — we're just giving it names.

Step 2 · The "aha"
The same task, two complexities

Here's the moment Big-O stops being abstract. Two ways to check "is this value in my collection?" — both correct, both common in real JS. But they scale completely differently.

// Version A — Array.includes()  →  O(n)
// To answer, JS scans the array element by element.
const seen = [1, 2, 3, ...]; // big array
seen.includes(target);    // worst case: look at ALL n items

// Version B — Set.has()  →  O(1)
// A Set is a hash table. It jumps straight to the answer.
const seen = new Set([1, 2, 3, ...]);
seen.has(target);         // one hop, regardless of size

On a small array you'd never notice. As n grows, the gap explodes. Don't take my word for it — run it. This searches for a value that isn't there (the true worst case) thousands of times in both an Array and a Set:

This is the whole game. Picking the data structure with the better Big-O is most of what "optimising" a solution means in interviews. Swapping an includes() (O(n)) inside a loop for a Set.has() (O(1)) is the move that turns an O(n²) brute force into a clean O(n) answer. You'll use this exact trick in the very next lesson.

Step 3 · The trap
A loop inside a loop = O(n²)

The most common complexity interviewers want you to escape. When you nest a loop over n inside another loop over n, you do n × n work:

// "Are there any duplicates?" — the brute-force way
for (let i = 0; i < arr.length; i++) {      // runs n times…
  for (let j = i + 1; j < arr.length; j++) { // …each doing ~n work  → O(n²)
    if (arr[i] === arr[j]) return true;
  }
}

Note: arr.includes() inside a map() or for loop is the sneaky version of this — a hidden loop inside a loop. Same O(n²) trap. The fix is almost always Step 2's idea: a Set or Map to make the inner lookup O(1).

Step 5 · Lock it in
The interview phrasing

Saying it out loud is a skill of its own. Practice this exact sentence shape on every problem from now on:

"This runs in O(___) time because [I loop over the input once / I have a nested loop / I do a constant number of lookups], and O(___) space because [I build a Map of size n / I only use a few variables]."
  • Big-O = how work grows with n; worst case; drop constants
  • One loop → O(n) · nested loop → O(n²) · direct lookup → O(1)
  • Set.has() / Map.get() are O(1) — the tool to kill nested loops
  • I can state both time and space complexity out loud
