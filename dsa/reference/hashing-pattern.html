The Hashing Pattern

Trade memory for speed: turn an inner search into an O(1) lookup.

When to reach for it — the trigger

If your brute force is "for each item, search the rest of the array" (a nested loop, O(n²)), a Map/Set almost always collapses the inner search to O(1), making the whole thing O(n).

Listen for these question shapes — each maps to a tool:

The questionToolStores
"Have I seen this before?" (duplicates, membership)Setjust the values
"How many times have I seen this?" (frequency, anagrams)Mapvalue → count
"Where did I see this / its partner?" (Two Sum)Mapvalue → index

The three template snippets

1 · Membership / duplicates

const seen = new Set();
for (const x of arr) {
  if (seen.has(x)) return true;   // duplicate found
  seen.add(x);
}

2 · Frequency count

const count = new Map();
for (const x of arr) {
  count.set(x, (count.get(x) ?? 0) + 1);
}

3 · Seen-value → index (Two Sum)

const seen = new Map();          // value → index
for (let i = 0; i < nums.length; i++) {
  const need = target - nums[i];
  if (seen.has(need)) return [seen.get(need), i];
  seen.set(nums[i], i);          // check BEFORE insert
}

Map vs Set vs plain object

UseWhen
SetYou only care whether something exists. Unique values.
MapYou need to associate each key with data (count, index). Keys can be any type; preserves insertion order.
{} objectQuick string-keyed counting. Fine, but Map is safer (no prototype-key surprises, any key type, has .size).

Under the hood — why O(1)

You'll almost never implement this in an interview, but you should be able to explain it — that's what signals real understanding, not just API recall:

StepWhat happens
1. HashA hash function turns the key into a number.
2. BucketThat number is reduced (% array size) to an index into an underlying array — the "bucket."
3. Store/lookupValue lives at that bucket. Lookup re-hashes the key and jumps straight there — no scanning, hence O(1).
4. CollisionsTwo keys can hash to the same bucket. Handled via chaining (a small list per bucket). Worst case (everything collides) degrades to O(n), but a good hash function makes this rare.
5. ResizeWhen too full, the table reallocates + rehashes everything — O(n) but rare, so it's amortized away and doesn't change average-case O(1).
Interview myth-buster: built-in Map/Set are expected and encouraged in standard coding interviews — using them well is the signal. You only implement a hash map from scratch if the question explicitly asks (e.g. LeetCode 706, "Design HashMap").

Complexity & the gotchas

AspectValue
TimeO(n) — one pass, O(1) per lookup/insert
SpaceO(n) — the Map/Set may hold up to n entries
Say it out loud: "O(n) time, O(n) space — I trade memory for speed." Naming the trade-off signals senior thinking in an interview.
Two Sum gotcha: check has(complement) before inserting the current value, or a number can falsely pair with itself.
