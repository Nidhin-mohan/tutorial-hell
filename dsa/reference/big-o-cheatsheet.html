What Big-O means in one line

Big-O describes how the work grows as the input n grows — not how many milliseconds it takes. We always quote the worst case, and we drop constants and lower-order terms (O(2n+5) → O(n)).

The complexity ladder (best → worst)

O(1)
O(log n)
O(n)
O(n log n)
O(n²)
O(2ⁿ)
Interview rule of thumb: if your brute force is O(n²), the interviewer almost always wants you to find an O(n) or O(n log n) way. That "better" approach is usually a pattern (hashing, two pointers, sliding window).

How to read complexity off your own code

You see…It's usually…
No loops, just a few operationsO(1)
One loop over the inputO(n)
A loop inside a loop, both over nO(n²)
Halving the problem each stepO(log n)
A sort, then a single loopO(n log n)
Building a Map/Set to avoid a nested loopO(n) time, O(n) space
Space complexity works the same way: count the extra memory you allocate that grows with n (a new array/Map of size n = O(n) space; a few variables = O(1)).
