Node.js Memory Leak Diagnosis

Quick reference — the four common leak shapes, the diagnostic method, and the tool decision. Companion to the memory-leak/ lab and Lesson 1.

The method (memorize this loop)

  1. Suspect a code path (an endpoint, a job, a connection type).
  2. Take a heap snapshot before.
  3. Exercise that code path under load for a while.
  4. Take a heap snapshot after.
  5. Compare the two — the diff is mostly what's leaking.

This is the canonical technique from the official Node.js diagnostics guide. Everything else below is refinement of this loop.

The four leak shapes

1. Unbounded cache

LEAK A Map/object used as a cache with no max size and no TTL. Often disguised as "just memoizing" — the bug is usually a cache key that's accidentally unique per call (a timestamp, a request id), which also guarantees a 0% hit rate.

FIX Stable, low-cardinality key. Bound the size (evict oldest/least-recently-used). Add a TTL and evict lazily on read.

Signature on a heap diff: one Map/object growing linearly with request count, holding many small, similar-shaped entries.

2. Dangling event listeners

LEAK emitter.on(event, handler) called once per connection/request with no matching .off()/.removeListener() when that connection ends.

FIX Remove the listener on the natural teardown event for that resource — for an HTTP response, that's res.on('close', () => emitter.off(event, handler)).

Signature: Node itself often tells you first — past 10 listeners on one event name, it logs MaxListenersExceededWarning: Possible EventEmitter memory leak detected. Don't silence this with setMaxListeners() before understanding why it fired.

3. Uncleared timers / closures

LEAK setInterval/setTimeout created per-entity (per request, per record) with the handle discarded — nothing can ever call clearInterval/clearTimeout on it. The timer's closure also keeps every variable it references alive for as long as the timer runs — i.e. forever.

FIX Store the handle (e.g. in a Map<id, Timeout>) and clear it from every code path that ends that entity's life: natural expiry, explicit delete, and error paths.

Signature: a growing count of active handles/timers that never shrinks, even long after the entities they were tracking are gone. process._getActiveHandles() (what clinic doctor samples) is a fast first check before reaching for heap snapshots.

4. Buffering instead of streaming

LEAK A handler materializes an entire response body (string or array) in memory before sending it. Not a permanent leak — it's a peak-memory-under-concurrency problem: N concurrent requests means N full buffers alive at once.

FIX Stream rows out as they're generated. In Node, the version that actually works is Readable.from(asyncGenerator) piped via stream/promises' pipeline() — not a plain for (...) res.write(row) loop.

Verified gotcha: a naive res.write() loop ignores backpressure — it queues every chunk into the socket's internal buffer regardless of whether the client is keeping up, and it blocks the event loop for the same duration as building one big string would. Measured in this lab: the naive loop used more peak RSS (~2GB) than the original buffering bug (~750MB) under a 15-concurrent burst. pipeline() is what actually bounds memory, because it only pulls the next chunk once the writable side signals it's ready.

Signature: sharp, traffic-correlated RSS spikes that recover after the burst — not a one-way climb. That shape (spike-and-recover) is what distinguishes a buffering problem from a true reference leak.

Tool decision

Heap snapshot diff (Chrome DevTools)

Best for: confirming what object is growing and tracing its retainer path back to the code that created it. Needs --inspect and a manual before/after comparison.

Chrome DevTools: Record heap snapshots

clinic doctor

Best for: a fast first-pass verdict on a running process, and for buffering-style problems (scenario 4) where the shape of the memory graph over time (climb vs. spike-and-recover) matters more than a single diff.

Glossary

RSS (Resident Set Size)
Total memory the process holds in physical RAM — heap plus native/C++ buffers plus stack. The number that answers "is this process actually using more memory."
heapUsed / heapTotal
V8's JavaScript object heap specifically. A leak in JS objects (caches, closures) shows up here; a leak in native buffers (large strings, streams) shows up more in external/arrayBuffers and RSS.
Retainer
In a heap snapshot, whatever is holding a reference to an object, preventing garbage collection. Tracing retainers backward from a leaked object is how you find the responsible code.
Backpressure
A stream signaling "I can't accept more data yet." Ignoring it (writing synchronously without checking the return value or waiting for 'drain') is how a "streaming" fix can still buffer everything in memory.
FIFO eviction
Removing the oldest inserted entry first when a bounded cache is full. Simple to implement with a Map, since it iterates in insertion order.
