Lesson 1 · Memory Leaks

Heap Snapshot Diffing

The one technique that underlies almost every memory leak diagnosis: take a snapshot before, exercise the suspect code, take a snapshot after, and read the difference.

Why this lesson exists: a real Node/Express service you maintain broke during a build, and the fix that shipped was "give it 3GB and move on." That works until it doesn't. This lesson is step one toward actually knowing what's growing and why instead of guessing at a bigger number.

The idea in one sentence

V8's garbage collector only frees memory that nothing references anymore — so a "leak" is never really about memory disappearing, it's about your code accidentally holding a reference to something it should have let go of. A heap snapshot is a complete map of every object alive at one instant, plus who's holding onto it. Two snapshots, compared, show you exactly what accumulated between them.

Walkthrough — diagnosing Scenario 1

The lab's memory-leak/src/scenario1-cache/leaky.ts caches a geo-IP lookup, but keys the cache with Date.now() — so it never hits, and every request adds a permanent entry. You're going to prove that with a snapshot diff instead of just trusting the README.

1Start the leaky server with the inspector on. 
cd memory-leak
npx tsx --inspect src/scenario1-cache/leaky.ts
Node prints a chrome://inspect-style URL. Open Chrome, go to chrome://inspect, and click inspect under Remote Target — this opens DevTools attached to your running server.
2Take the "before" snapshot. In DevTools, go to the Memory tab, select Heap snapshot, and click Take snapshot. Label it mentally as "before" — you don't need to touch the endpoint yet.
3Exercise the suspect code path. In a second terminal, note the demo link's shortCode printed on startup, then drive load against it with varying IPs so the cache actually gets exercised: 
npm run load-test -- "http://localhost:3000/links/<shortCode>/geo?ip=10.0.{i}.1" 2000 20 0 500
4Take the "after" snapshot. Back in DevTools, take a second heap snapshot once the load test finishes.
5Compare them. Select the second (newer) snapshot in the left panel, then switch the dropdown from Summary to Comparison. Sort by Size Delta, descending. You're looking for the constructor/object type with the largest positive delta — in this case, that should be the Map backing geoCache, holding roughly one entry per request you fired.

That's the whole loop. Everything else — clinic doctor, production APM graphs, process.memoryUsage() polling — is either a faster first-pass signal that tells you where to point this technique, or a coarser version of the same before/after comparison.

Timers are real JS objects with real closures — they absolutely show up. Timeout handles and the functions/variables they close over are exactly the kind of thing a heap snapshot is good at surfacing.

Try it against the fixed version

Repeat steps 1–5 against npm run scenario1:fixed with the same load test. The Comparison view should show the Map's size delta plateauing — new entries stop growing past the 500-entry cap once the FIFO eviction kicks in. Seeing that plateau in the diff itself, not just in the /memory endpoint's printed numbers, is the actual skill this lesson is building.
